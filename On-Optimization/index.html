A Better way to 'optimize

Imagine this, you have a scene with thousands of little points joined together that we call vertices. This scene has various models, textures, lighting, etc. When things especially get complex, you start to realize that your GPU is only so powerful.. So, you implement a 'fix'. This 'fix' is what's known as frustum calling. The basic gist of it is, you only render what you can see. By default, when this isn't implemented, you are usually rendering the entire world around you even what you cannot see. This is fine but quickly becomes a problem when aiming to build especially complex scenary. So, you trim out the fatigue and stop the world from rendering around you, but only what's in front of you.

Well, today I say screw that! I am introducing a new thereotical model that when implemented, could actually be revolutionary. Now, if you're particular fundamentalists, you can see the issue here. This Solution I described in the above paragraph has a core issue and it's that the world is number #1 still technically rendering. You have a constant loop in the background that is constantly commanding the Engine to only render what is in view.

And while this is okay, especially if you're CPU is particularly fast, it's inefficient. The Vertices that are in your GPU are ultimately still loaded even when not being rendered. My algorithm comes down to this: "Not only render, but only use what is needed.". The Engine still 'uses' the vertices for a "just in case" scenario when the player turns directions. My implementation will not only not Render the vertices, but the engine won't even load the vertices in the GPU. This make's more room for what is infront of the player.

If done right, this could increase the performance to absurd levels. Only load what is needed.
